# Tasks — Ways of Working repository improvements

Derived from `plan.md` (read it first for context and rationale). Tasks are ordered by phase; within a phase, order is a suggestion unless a dependency is noted. Each task states its acceptance criteria. The book's spec at `doc/digital-health-ways-of-working/spec/index.md` is the source of truth for anything book-related — where a task and the spec disagree, the spec wins (§14: or change the spec deliberately in the same commit).

Conventions for the executing agent:

- Work in small commits, one task (or one coherent group) per commit; reference the task ID (e.g. `[C1]`) in the commit message.
- Never invent citations, URLs, statistics, or quotations (spec §6). Verify every Wikipedia slug before linking.
- UK spelling in the book; keep the historical articles under `doc/*` verbatim except navigation/typo fixes.
- After Phase 2 lands, run `make check` before every commit.

## Phase 2 — Validation & automation (Workstream A, G)

- [ ] **A0. Scaffold `tools/` + `Makefile`.**
  - Create `tools/` for validator scripts (Python 3, stdlib only — no pip dependencies) and a root `Makefile` with targets: `check` (all gates), `check-structure`, `check-consistency`, `check-links`, `check-refmatter`, `check-repo-links`, `spell` (optional, warns only).
  - Shared module `tools/manifest.py`: parses spec §4 into a canonical list of (part, number, title, filename); reused by every other tool and by Phase 3 builds.
  - Accept: `make check` runs all gates and exits non-zero on any failure; manifest parser round-trips all 64 chapters + preface.

- [ ] **A1. Structural gate checker (`tools/check_structure.py`).** Per spec §3/§10, for every `PP-CC-*.md`:
  - 13 sections present, correctly named, in order; first body line is a single bold sentence; `## Best practices` numbered with bold lead-ins; exactly three bold numbered questions each followed by a 4–8 sentence briefing, placed before `## In practice`; `### Startup` / `### Enterprise` / `### Government` in that order; maturity table has the four columns Initial/Developing/Defined/Optimising and 3–5 rows; checklist has 6–12 `- [ ]` items.
  - Accept: current 64 chapters pass (fix any real violations found — report them, don't silently loosen the checker); deliberately broken test fixture fails.

- [ ] **A2. Consistency gate checker (`tools/check_consistency.py`).**
  - Filename ↔ `# Chapter N — Title` ↔ spec manifest ↔ book `README.md` ToC agreement.
  - Every in-prose `Chapter N — Title` cross-reference matches the manifest's number→title mapping.
  - Word-count report: flag chapters under 3,000 substantive words (warning, not failure).
  - Accept: all cross-references in the book currently resolve, or a fix-list is produced and fixed.

- [ ] **A3. Link gate checker (`tools/check_links.py`).**
  - Per chapter: extract inline Wikipedia links; 5–12 of them; each also present in `## References`; no duplicate slugs pointing at different anchor texts for the same concept.
  - `--online` flag: HEAD-request each Wikipedia URL with a local cache file; default is offline (CI runs offline; a scheduled weekly CI job runs `--online`).
  - Accept: offline checks pass on current book; online run produces a report of any 404 slugs (fix any found).

- [ ] **A4. Reference-matter checker (`tools/check_refmatter.py`).**
  - GLOSSARY.md: alphabetical within letter sections; every "See Chapter N" points to a manifest chapter; every Wikipedia link well-formed.
  - INDEX.md: same, plus numbers ascending and de-duplicated per entry.
  - Accept: passes after C1; fails on the pre-C1 empty INDEX.md (i.e. it must check non-emptiness).

- [ ] **A5. Repo-wide relative-link checker (`tools/check_repo_links.py`).**
  - Walk every `.md` in the repo; verify each relative link target exists. Whitelist external URLs (don't fetch).
  - Accept: repo passes after D1's link fixes.

- [ ] **A6. Spell-check configuration.**
  - Expand `cspell.json`: enable `en-GB`, add a project dictionary file (`.cspell/project-words.txt`) seeded with NHS/health/digital vocabulary (DTAC, Caldicott, interoperability terms, ICS, NICE, MHRA, proper nouns from the book) discovered by an initial `cspell` run.
  - Accept: `make spell` runs with < 50 residual warnings across the book, none of which are real misspellings left unfixed (fix real typos it finds).

- [ ] **A7. GitHub Actions CI (`.github/workflows/ci.yml`).**
  - On push/PR: checkout, `make check`. Separate scheduled weekly job: `make check-links ONLINE=1`.
  - Add CI badge to root README.
  - Accept: workflow YAML is valid; `make check` green locally (CI verification happens on first push).

- [ ] **G1. Hygiene.** `.gitignore` (add `dist/`), `.editorconfig` (Markdown: UTF-8, LF, no trailing whitespace enforcement on `.md` since some files may embed hard breaks).
  - Accept: files present; `git status` clean after a `make book` run (Phase 3).

- [ ] **G2. Book versioning.** Add a `## Version history` (or `VERSION` line) to the spec per §14's "versioned events" language, seeded at `1.0.0` for the current 64-chapter structure, with a one-line policy: structural change = minor bump, chapter rewrite = patch note.
  - Accept: spec updated in a single self-contained commit; policy is ≤5 lines.

  - Cover at minimum the book; ideally also the `doc/*` articles as a second section.
  - Add a `.github/workflows/pages.yml` deploying to GitHub Pages on push to main (guard: only if Pages is enabled; otherwise document the one-click enable step in `tools/README.md`).
  - Accept: `make site` builds locally without errors; nav lists all 64 chapters in order; root README links to the (future) Pages URL.

- [ ] **B4. Prev/next chapter navigation (`make nav`).**
  - `tools/build_nav.py` inserts/updates a single generated footer line in every chapter between markers (e.g. `<!-- nav -->…<!-- /nav -->`): previous chapter, contents link, next chapter — titles included per the spec's cross-reference rule.

  - Accept: all 64 chapters + preface have correct nav; `make nav && git diff --quiet` after second run.

- [ ] **B6. `llms-full.txt` generation.** Fold the old llms.txt's full-content role into the build: `make llms` regenerates `llms.txt` (curated index, kept in git) and `dist/llms-full.txt` (concatenation).
